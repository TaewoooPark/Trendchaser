# Trendchaser 2026-05-04 업데이트

**한 줄 요약 — "오래된 trending이 brief를 잠식하던 패턴"을 코드·점수·소스 세 단계에서 동시에 끊었다. 그 자리에 release.atom firehose와 foundation lab 보강이 들어왔다.**

> 동기: 최근 brief 10건을 점검한 결과 GitHub trending 항목이 평균 25%, aggregator 항목이 33%를 차지하며 "1차 발신처가 직접 push한 신호"가 밀려나는 구조적 편향을 확인. trending은 본질적으로 "이미 존재하던 것의 재부각"이므로 freshness 신호로 보기 어렵다.

---

## 1. 점수화 — novelty axis 도입 (가장 큰 변화)

`prompts/curate.md` Step 6를 5-axis → **6-axis로 확장**:

```
변경 전:  signal 30 + affinity 30 + recency 25 + velocity 10 + freshness 5
변경 후:  signal 25 + affinity 25 + recency 20 + novelty 20 + velocity 5 + freshness 5
```

**novelty의 정의**:

| 소스 클래스 | novelty 점수 |
|---|---|
| 1차 발신처 (lab 블로그·release.atom·hf_papers·hn_breaking·x_lab) | 100 |
| 큐레이터 / aggregator (simonwillison, hn_top, techmeme 등) | 65 |
| 트렌딩 (이미 존재하던 artifact의 재부각) — `created_at` 기준 | ≤7d 80 / ≤30d 50 / ≤90d 25 / >90d 0 |
| arxiv_ai (후행 raw DB) | 50 |

이 axis가 "GitHub trending 1위지만 사실 2년 전 repo가 단발 hype를 받는 것"과 "방금 nvidia가 release를 push한 것"을 정량적으로 구분한다 — 트렌딩 1.0× / 1차 발신 1.6× 효과.

## 2. 트렌딩 cap — 구조적 차단

`scripts/shortlist.py`에 hard cap 추가. Step 6 점수와 무관하게 코드 단계에서 강제:

- AI 섹션: 트렌딩 소스 최대 **2개**
- General 섹션: 트렌딩 소스 최대 **1개**

`scripts/enrich.py`에는 추가로 **novelty multiplier**가 prelim score에 곱해진다 (트렌딩 소스 한정, `created_at` 기준 ≤7d 1.3× / ≤30d 1.0× / ≤90d 0.7× / >90d 0.4×). 이 곱셈자는 enrichment 대상 top-30 선택 단계에서 적용되므로, 오래된 trending repo는 본문 추출 자원도 못 받는다.

## 3. 새로 잡는 신호

### 3-1. GitHub release.atom firehose 14개 신설

GitHub `releases.atom`은 tag push 직후 갱신되는 사실상 실시간 피드 — release 이벤트가 1차 발신되는 가장 빠른 자리다. 이번 개정에서 LLM 인프라·에이전트 프레임워크·MCP SDK를 한꺼번에 보강:

```
openai-python, openai-agents-python, openai/swarm,
huggingface/transformers, accelerate, diffusers, peft,
vllm-project/vllm, ollama/ollama, ggml-org/llama.cpp,
langchain-ai/langchain, microsoft/autogen,
modelcontextprotocol/python-sdk, modelcontextprotocol/typescript-sdk
```

기존 `claude_code_releases`, `mcp_releases`와 함께 16개 release 채널이 작동한다. weight 1.5–1.6.

### 3-2. `hf_models_new` 신설 + `hf_models_trending` 강등

| ID | 이전 | 현재 |
|---|---|---|
| `hf_models_trending` | weight 1.3, sort=likes7d | weight **0.9**, mode=trending (역할 분담) |
| `hf_models_new` | (없음) | weight **1.5**, mode=new |

`hf_models_new`는 whitelist의 각 foundation lab을 author 필터로 순회하며 `createdAt desc`로 최신 N개를 받아 7일 윈도우로 컷한다. 글로벌 createdAt sort는 무명 사용자 업로드에 묻혀 foundation lab을 못 잡는 게 측정으로 확인됨(top-1000 중 0건). per-author 순회로 확실히 잡는 방식.

### 3-3. `hn_breaking` — Hacker News 4시간 firehose

기존 `hn_top`은 누적 점수 기반(`/api/v1/search`, lookback 30h, 120pt 임계). 이미 화제가 된 스토리를 잡는다.

신규 `hn_breaking`은 `/api/v1/search_by_date` 기반 4시간 윈도우. AI 키워드 매치 시 75pt 임계. **"곧 화제가 될 스토리의 초기 모멘텀"** 을 잡는 firehose. 두 채널이 시간 축에서 직교한다.

## 5. X 채널 가중치 상향

직접 큐레이션한 follow 리스트 채널 weight를 **1.5 → 1.7**로 상향. 자체 호스팅 mini-relay를 거쳐 raw 텍스트만 수집하는 슬롯 스냅샷 방식이라 라이브 firehose는 아니지만, lab 발표가 블로그보다 X에 먼저 뜨는 케이스 일부를 메운다 — follow 리스트 자체가 노이즈 필터 역할을 하므로 SNR이 높다는 점이 weight 상향의 근거.

## 6. brief 톤 지침 정리

내부 운영 메모로 간직: brief는 신문 뉴스 톤이며 외부 독자 대상 산문 — 1·2인칭 직접 지칭("귀하의 follow", "사용자가 최근 본") 금지, 파이프라인 메타("운영 메모", "직접 수집한") 노출 금지. brief 본문은 사건·사실 진술로만 구성.
