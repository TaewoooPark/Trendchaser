# Trendchaser 2026-05-04 업데이트

**한 줄 요약 — brief에 "어제 공개된 신호"가 더 자주 올라옵니다. 그동안 트렌딩이 차지하던 자리를 lab release, 모델 업로드, breaking news 같은 1차 발신처가 채우도록 구조를 바꿨습니다.**

이번 개정의 방향은 단순합니다. "이미 화제인 것"보다 "방금 발표된 것"을 더 빨리, 더 많이 포착하는 데 초점을 맞췄습니다. 이를 위해 새로운 채널을 추가하고, 점수화 기준을 다듬고, 오래된 트렌딩 항목이 첫 줄을 점령하지 못하도록 구조적으로 차단했습니다.

## 무엇이 늘어났는가

총 16개의 신규 채널과 31개 foundation lab의 추가 커버리지를 확보했습니다. 그 결과 brief의 첫 줄이 "공개된 지 24시간이 채 지나지 않은 항목"이 될 확률이 크게 높아졌습니다.

### 새 경로 1 — GitHub Release Firehose (14개 신설)

GitHub에서 모델·프레임워크가 새 버전을 태그하는 순간, 가장 빠르게 갱신되는 신호가 각 repo의 `releases.atom`입니다. 이번 개정에서는 LLM 인프라, 에이전트 프레임워크, MCP SDK를 한꺼번에 신설했습니다.

| 카테고리 | 채널 |
|---|---|
| OpenAI 생태계 | openai-python, openai-agents-python, openai/swarm |
| HuggingFace 스택 | transformers, accelerate, diffusers, peft |
| 추론 엔진 | vllm, ollama, llama.cpp |
| 에이전트 프레임워크 | langchain, autogen |
| MCP SDK | python-sdk, typescript-sdk |

기존 `claude_code_releases`, `mcp_releases`까지 합치면 **release 채널이 총 16개**로 늘어납니다. 가중치(weight 1.5–1.6) 덕분에 vllm이 새 버전을 릴리스하거나 transformers가 새 모델 패밀리를 머지하는 순간, 그 변화가 다음 brief에 곧바로 반영됩니다.

### 새 경로 2 — HuggingFace 신규 업로드 추적

기존 `hf_models_trending`은 "이미 인기 있는 모델"을 다룹니다. 그래서 lab이 모델을 막 올린 시점에는 잘 포착되지 않는 한계가 있었습니다.

신설된 `hf_models_new`는 **whitelist된 47개 lab**(meta-llama, google-deepmind, nvidia, mistralai, deepseek-ai, Qwen 등 + 이번에 보강된 31개)을 한 곳씩 순회하면서, 최근 7일 안에 올라온 모델을 수집합니다. 동작 예시는 다음과 같습니다.

- 어제 Qwen이 새 7B 모델을 업로드 → 다음 slot brief 후보
- 지난주 BlackForestLabs가 Flux 후속 모델을 공개 → 7일 윈도우 안에 들어 후보

기존 trending 채널의 가중치는 1.3에서 0.9로 낮춰, 두 채널의 역할을 명확히 분리했습니다.

### 새 경로 3 — Hacker News Breaking

기존 `hn_top`은 누적 점수 기준으로 "이미 화제가 된 글"을 가져옵니다. 신설된 `hn_breaking`은 **최근 4시간 안에 올라온 글 중 AI 키워드가 매치되면서 75점 이상**인 항목을 추적합니다.

```
hn_top      = "지금 HN 첫 페이지에 있는 것"
hn_breaking = "곧 첫 페이지에 올라갈 것"
```

두 채널이 서로 다른 시간대를 담당하면서 보완 관계를 이룹니다.

## 점수화 기준 변경 — Novelty 축 도입

기존 5개 축에 **novelty(참신성)** 축을 추가해 6축으로 확장했습니다.

| 축 | 이전 | 현재 |
|---|---|---|
| signal | 30% | 25% |
| affinity | 30% | 25% |
| recency | 25% | 20% |
| **novelty** | — | **20% (신규)** |
| velocity | 10% | 5% |
| freshness | 5% | 5% |

novelty 점수는 다음 기준으로 부여됩니다.

| 소스 유형 | novelty 점수 |
|---|---|
| 1차 발신처 (lab 블로그, release, papers) | **100** |
| 큐레이터 / aggregator | 65 |
| 트렌딩 (created 7일 이내) | 80 |
| 트렌딩 (created 30일 이내) | 50 |
| 트렌딩 (created 90일 이내) | 25 |
| 트렌딩 (90일 초과) | 0 |
| arxiv 원문 raw | 50 |

이 축은 "GitHub trending 1위에 올라 있지만 실제로는 2년 전 repo가 단발성 hype를 받는 항목"과 "방금 nvidia가 release한 항목"을 정량적으로 구분합니다. 같은 트렌딩이라도 더 신선한 쪽이 brief 상단에 자리잡습니다.

## 트렌딩 한도 명시

코드 단계에서 강제되는 hard cap을 새로 도입했습니다.

- **AI 섹션**: 트렌딩 소스 최대 2개
- **General 섹션**: 트렌딩 소스 최대 1개

이를 통해 "trending이 brief 절반을 차지하는" 패턴이 구조적으로 차단됩니다. 여기에 더해, 오래된 trending repo는 본문 추출 자원조차 받지 못하도록 enrichment 단계에서도 신선도 가중치(≤7d 1.3× / >90d 0.4×)가 함께 곱해집니다.
