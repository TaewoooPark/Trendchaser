# Trendchaser 2026-05-04 업데이트

**한 줄 요약 — brief에 "어제 push된 신호"가 더 자주 올라온다. 트렌딩이 자리를 차지하던 분량이 1차 발신처(lab release, 모델 업로드, breaking news)로 교체됐다.**

이번 개정의 방향은 단순하다. "이미 화제인 것"보다 "방금 발표된 것"을 더 빨리, 더 많이 잡는다. 그러기 위해 새 채널을 트고, 점수화 기준을 바꾸고, 오래된 트렌딩이 첫 줄을 점령하지 못하도록 막았다.

---

## 무엇이 늘어났나

총 **16개 신규 채널** + **31개 foundation lab** 추가 커버리지. brief의 첫 줄이 "release된 지 24시간 안 된 무언가"가 될 확률이 크게 높아진다.

### 새 경로: GitHub Release Firehose (14개)

GitHub에서 모델·프레임워크가 새 버전을 태그하는 순간, 그 시점에 가장 빠르게 갱신되는 게 각 repo의 `releases.atom`이다. 이번 개정에서 LLM 인프라·에이전트 프레임워크·MCP SDK를 한꺼번에 신설:

| 카테고리 | 채널 |
|---|---|
| OpenAI 생태계 | openai-python, openai-agents-python, openai/swarm |
| HuggingFace 스택 | transformers, accelerate, diffusers, peft |
| 추론 엔진 | vllm, ollama, llama.cpp |
| 에이전트 프레임워크 | langchain, autogen |
| MCP SDK | python-sdk, typescript-sdk |

기존 `claude_code_releases`, `mcp_releases`까지 합쳐 **release 채널이 16개**가 됐다. 새 모델 가중치(weight 1.5–1.6)로 체감되는 효과는 — vllm이 0.x.y를 찍는 순간, 또는 transformers가 새 모델 패밀리를 머지하는 순간이 다음 brief에 뜬다.

### 새 경로: HuggingFace 신규 업로드 추적

기존 `hf_models_trending`은 "이미 인기 있는 모델"을 잡는다. 그래서 lab이 모델을 막 올렸을 때는 잘 안 잡힌다.

신설 `hf_models_new`는 **whitelist된 47개 lab**(meta-llama, google-deepmind, nvidia, mistralai, deepseek-ai, Qwen 등 + 이번에 보강된 31개)을 한 곳씩 순회하며 최근 7일 안에 올라온 모델을 잡는다. 즉:

- 어제 Qwen이 새 7B 모델을 올림 → 다음 slot brief 후보
- 지난주 BlackForestLabs가 Flux 후속 모델을 푸시함 → 7일 윈도우 안이라 후보

기존 trending 채널은 weight를 1.3 → 0.9로 낮추고 역할을 분리했다.

### 새 경로: Hacker News Breaking

기존 `hn_top`은 누적 점수 기준(이미 화제가 된 글). 신설 `hn_breaking`은 **최근 4시간 안에 올라온 글 중 AI 키워드가 매치되고 75점 이상**을 잡는다.

```
hn_top      = "지금 HN 첫 페이지에 있는 것"
hn_breaking = "곧 첫 페이지에 올라갈 것"
```

두 채널이 시간 축에서 직교한다.

직접 follow 리스트를 큐레이션해 둔 X 채널의 가중치를 **1.5 → 1.7**로 올렸다. lab 발표가 블로그보다 X에 먼저 뜨는 케이스를 더 적극적으로 잡기 위함. follow 리스트 자체가 노이즈 필터 역할을 하므로 신호 대비 잡음이 낮다는 점이 근거.

## 점수화 기준 변경 — Novelty 축 도입

기존 5축에 **novelty(참신성)** 축을 추가해 6축으로 확장:

| 축 | 이전 | 현재 |
|---|---|---|
| signal | 30% | 25% |
| affinity | 30% | 25% |
| recency | 25% | 20% |
| **novelty** | — | **20% (신규)** |
| velocity | 10% | 5% |
| freshness | 5% | 5% |

novelty는 다음 기준으로 매겨진다:

| 소스 유형 | novelty 점수 |
|---|---|
| 1차 발신처 (lab 블로그, release, papers) | **100** |
| 큐레이터 / aggregator | 65 |
| 트렌딩 (created 7일 이내) | 80 |
| 트렌딩 (created 30일 이내) | 50 |
| 트렌딩 (created 90일 이내) | 25 |
| 트렌딩 (90일 초과) | 0 |
| arxiv 원문 raw | 50 |

이 축이 "GitHub trending 1위지만 사실 2년 전 repo가 단발 hype를 받는 것"과 "방금 nvidia가 release를 push한 것"을 정량적으로 구분한다. 같은 트렌딩이라도 신선한 게 brief에 잘 올라간다.

## 트렌딩 한도 명시

코드 단계에서 강제되는 hard cap을 새로 뒀다:

- **AI 섹션**: 트렌딩 소스 최대 2개
- **General 섹션**: 트렌딩 소스 최대 1개

이로써 "trending이 brief 절반을 차지하는" 패턴이 구조적으로 막힌다. 추가로 오래된 trending repo는 본문 추출 자원조차 받지 못하도록 enrichment 단계에서도 신선도 가중치(≤7d 1.3× / >90d 0.4×)가 곱해진다.

## 다음 슬롯부터 체감되는 변화

| 항목 | 이전 | 이후 |
|---|---|---|
| brief 1번 항목이 "어제 release된 무언가"일 확률 | 낮음 | **높아짐** |
| brief에 "2년 전 repo가 오늘 trending" 항목 등장 | 평균 25% | ~10% 이하 |
| Foundation lab 신모델 발표 포착 지연 | 종종 누락 | 평균 1슬롯 내 포착 |

요약하면 — **brief가 "지금 막 일어나는 일"에 더 가까워진다.** 트렌딩 항목이 사라지는 게 아니라, 트렌딩의 자리를 절반 이상 1차 발신처가 차지하게 된다.
