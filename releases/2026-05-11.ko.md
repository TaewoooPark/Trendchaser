## 무엇이 고쳐졌나 (사용자 체감 포인트)

### 패치 1 — Anthropic 글이 매번 "방금 공개"로 끌려오던 문제 종료

**체감되던 현상**: 일주일 혹은 2주 전에 올라온 Anthropic 글이 brief의 첫 항목에 "방금 발표된 것처럼" 다시 등장하던 패턴.

**원인**: Anthropic 사이트는 sitemap을 사용하는데, sitemap의 `lastmod` 값은 **글의 실제 발행 시점이 아니라 사이트 자체가 재배포될 때마다 갱신**됩니다. 그래서 Anthropic이 사이트를 한 번 재배포하면 모든 글의 `lastmod`가 한꺼번에 새 시각으로 바뀌고, 파이프라인이 이를 "전부 방금 공개"로 읽어버렸습니다.

**조치**: 페이지를 가져올 때 메타데이터에서 실제 발행 시점을 추출합니다 — `og:article:published_time` → JSON-LD `datePublished` 순으로 우선 적용. sitemap `lastmod`는 보조 수단으로만 쓰이고, 실제 발행시점과 30일 이상 차이가 나면 "재발행"으로 표시됩니다.

같은 패턴(sitemap `lastmod` ≠ 실제 발행시점)은 Anthropic뿐 아니라 모든 sitemap 기반 소스에 일괄 적용됩니다.

### 패치 2 — Release atom의 알파·릴리스 후보·의존성 갱신 도배 차단

**체감되던 현상**: "claude-code v2.3.4-rc.1 출시" 같은 사전 릴리스나 transformers 패치 버전 갱신이 brief 자리를 잠식하던 패턴.

**원인**: GitHub의 `releases.atom`은 **모든 태그**를 내보냅니다 — 사전 릴리스, 패치 버전 갱신, 의존성 업데이트 커밋 모두 포함됩니다. 직전 필터(X.Y.0 메이저/마이너만 통과)는 너무 거칠어서, semver 규칙을 엄격히 따르지 않는 정당한 release(`llama.cpp b9012`, `MCP 2025-11-25`, `langchain 1.4.3` 등)까지 통째로 막아버리는 부작용이 있었습니다.

**조치**: 3단 release 분류기로 교체했습니다.

- **사전 릴리스** (`-rc`, `-alpha`, `-beta`) → 제거
- **메이저 / 마이너** (X.Y.0) → 통과
- **패치 / semver 비준수** → 본문에 실질 신호어(`new model`, `breaking change`, `CVE-`, `new API` 등)가 매치될 때에만 통과

따라서 패치 버전 갱신도 실제 뉴스 가치가 있을 때(보안 패치, 새 모델 패밀리 추가 등)는 살아남고, 단순 의존성 갱신 잡음만 걸러집니다. `llama.cpp`나 MCP 명세 날짜처럼 semver 규칙을 따르지 않는 1차 발신은 누락 없이 그대로 흐릅니다.

### 패치 3 — HuggingFace 트렌딩이 "예전 모델, 갑자기 인기"를 "방금 업로드"로 오인하지 않도록

**체감되던 현상**: brief가 가끔 몇 달 전에 올라온 HuggingFace 모델을 첫 줄에 올리던 패턴 — 트렌딩 점수 급등만으로.

**원인**: `hf_models_trending`이 업로드 날짜만 보다 보니, 모델 가중치 갱신 푸시(6개월 전 모델에 새 가중치가 올라가는 정당한 신선 신호)를 구분하지 못했습니다.

**조치**: 트렌딩 후보 상위에 한해 모델 상세 정보를 가져와 `lastModified`를 확인합니다. 갓 업로드된 모델이면 신규 출시로, 예전 모델이지만 최근 가중치 갱신이 있었으면 "revision push" 이벤트로 표기합니다 — 둘은 다르게 처리됩니다. 둘 다 아닌 경우(트렌딩 알고리즘이 단지 다시 발견한 경우)는 더 낮은 트렌딩 novelty 점수를 받습니다.

### 패치 4 — Aggregator(Techmeme, Lobsters 등)가 2주 묵은 글을 fresh로 세탁하지 못하게

**체감되던 현상**: 2주 전 글이 brief에 "방금 공개"인 것처럼 등장하고, 출처가 Techmeme이나 Lobsters AI로 표시되던 패턴.

**원인**: aggregator 피드의 `pubDate`는 **그 aggregator가 링크를 큐레이션한 시점**을 나타내지, 원문이 발행된 시점이 아닙니다. 파이프라인이 그 aggregator pubDate를 "원문 발행시점"으로 그대로 받아 처리하고 있었습니다.

**조치**: aggregator 피드(`techmeme`, `lobsters_ai`, 그리고 이번에 비활성된 `producthunt`)는 이제 "신호용"으로만 취급됩니다 — aggregator는 한 글에 주의를 환기시킬 수 있지만, 발행 시점은 비워두고 원문에서 다시 추출합니다. 더는 2주 묵은 글이 "방금 공개"로 둔갑하지 않습니다.

`producthunt`는 같은 패스에서 비활성 처리했습니다. pubDate 불일치 문제에 더해, 일반적으로 신호 대비 잡음 비율이 낮습니다(런칭 대다수가 brief가 다루는 종류의 신호를 담지 않습니다).

### 패치 5 — X 채널 신호 복구, 사이클 시간 절반으로

**체감되던 현상(직전 며칠)**: brief에 X 출처 항목이 거의 없었습니다. follow 리스트 X 채널이 사실상 침묵.

**원인**: 클라우드의 routine 환경 변수 두 개(`VERCEL_PROXY_BASE`, `PRISMA_VERCEL_SECRET`) — X 캐시 워밍 프록시를 가리키는 — 가 누락되어 있었습니다. 이 변수가 없으면 X fetcher는 빈 리스트를 조용히 반환합니다(에러 없음, 단지 0건).

**조치**: routine 환경에 변수를 등록했습니다. 로컬 검증 결과 경로가 정상 복구되었습니다(메인 follow-list 채널이 슬롯당 정상적인 양의 항목을 반환).

**부수 효과 — 사이클 시간 138분 → 약 70분**: X 미니 프록시가 매 사이클마다 follow 리스트 83계정 전부를 예열하고 있었는데, 그중에는 몇 달간 트윗이 없는 휴면 핸들(`@arena`, `@steipete`, `@ssi`, `@ilyasut`, `@lilianweng` 등)이 다수 섞여 있었습니다. 각 휴면 핸들이 12초의 사이클 예산을 헛소비하던 셈. 이제 프록시가 **죽은 핸들을 추적**합니다 — 3회 연속 빈 응답이면 24시간 동안 휴면으로 표시하고, 한 번이라도 살아있는 응답이 오면 즉시 해제. 한 cron 주기마다 처리하는 묶음 크기를 5에서 7로 올린 것과 합쳐, 풀 사이클 지연이 **138분 → 약 70분**으로 줄었습니다. 배포 후 약 6시간 동안 휴면 목록이 채워지면서 안정화됩니다.

체감: 연구소 공식 계정이 트윗을 올리면 brief가 약 140분이 아니라 **약 70분 안에** 그 트윗을 봅니다.

### 패치 6 — 끊긴 1차 발신처 RSS 교체

AI 연구소 두 곳이 공지 없이 블로그 RSS 피드를 내렸습니다. URL이 모든 후보 경로에서 404를 반환하기 시작했습니다.

| 연구소 | 끊긴 피드 | 상태 | 대체 |
|---|---|---|---|
| Meta AI | `ai.meta.com/blog/rss/` | 404 | `meta-llama/llama-stack` releases (가중치 1.5) |
| Mistral | `mistral.ai/news/feed.xml` | 404 | `mistralai/client-python` (1.5) + `mistralai/mistral-common` (1.4) |

대체로 들어온 GitHub release atom은 기존 16개 release 피드와 같은 firehose 표면입니다. 라이브 검증 완료(Meta llama-stack v0.8.0 / 2026-05-01, Mistral client-python v2.4.0 / 2026-04-16, mistral-common v1.11.0 / 2026-04-29). Meta나 Mistral가 공식 도구의 새 버전을 푸시하면 brief가 한 슬롯 안에 그것을 포착합니다.
