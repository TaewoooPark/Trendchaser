# Trendchaser 2026-05-11 update

**TL;DR — the brief stops mistaking old articles for "just published." Three regressions that flooded the top of the brief with stale-but-fresh-looking items are now sealed off, two dead first-party RSS sources have been swapped for live release atoms, and X channels — silent for several days — start flowing again.**

This update is mostly about **freshness honesty**. The previous build had several silent failure modes where an article published two weeks ago could surface as if it had just dropped. Each of those paths has now been closed. Separately, two RSS feeds that AI labs quietly retired (Meta AI, Mistral) have been replaced with their official GitHub release atoms — so the signal pipe is whole again. And the X side, which was mute for several days due to a misconfiguration, now flows with cycle time cut roughly in half.

## What's been fixed (user-perceptible)

### Fix 1 — Anthropic articles no longer keep showing up "fresh"

**Symptom you might have noticed**: every now and then, an Anthropic article from a week or two ago would re-appear as the lead item, as if it had just been posted.

**Root cause**: the Anthropic site uses a sitemap, and that sitemap's `lastmod` field updates whenever the site itself is redeployed — not when the article is actually published. So whenever Anthropic shipped a site redeploy, every article's lastmod ticked forward, and the pipeline read that as "just published."

**Fix**: when fetching the page, the pipeline now extracts the real publish timestamp from the page's metadata (`og:article:published_time` or JSON-LD `datePublished`). Sitemap `lastmod` is now only a fallback, and if it disagrees with the real publish time by more than 30 days, the article is flagged as a republish rather than a fresh post.

The same pattern (sitemap `lastmod` ≠ real publish) is now blocked in general, not just for Anthropic.

---

### Fix 2 — Release atoms stop flooding the brief with alpha / rc / dependency-bump churn

**Symptom**: items like "claude-code v2.3.4-rc.1 released" or "transformers patch bump" filled space in the brief that should have gone to actual headline news.

**Root cause**: GitHub's `releases.atom` feed broadcasts every tag — including pre-releases, patch bumps, and dependency-update commits. The previous filter (X.Y.0-only) was too crude: it accidentally killed legitimate releases that don't follow strict semver, like `llama.cpp b9012` or `MCP 2025-11-25`.

**Fix**: a 3-tier release classifier now runs:
- **Pre-release** (`-rc`, `-alpha`, `-beta`) → drop
- **Major / minor** (X.Y.0) → pass through
- **Patch / non-semver** → only pass if the body mentions a substantive signal (`new model`, `breaking change`, `CVE-`, `new API`, etc.)

So patch bumps still surface when they actually carry news (a CVE fix, a new model family), but routine dependency-bump churn is muted. Non-semver release lines like `llama.cpp` or MCP spec dates remain in the firehose with no false negatives.

### Fix 3 — HuggingFace trending stops mistaking "old model, suddenly popular" for "just uploaded"

**Symptom**: the brief sometimes led with a HuggingFace model that was actually months old, just because it had a sudden spike in trending.

**Root cause**: `hf_models_trending` was using the upload date alone, which doesn't account for model revision pushes (a 6-month-old model getting a fresh weights push is genuinely fresh signal).

**Fix**: for the top trending candidates, the pipeline now fetches model detail and looks at `lastModified`. If the model is freshly uploaded (createdAt ≈ lastModified), it surfaces as a brand-new release. If the model is old but had a recent revision push, it's tagged as a "revision push" event so you can tell it apart from a true new release. If neither applies — i.e. the model is just being re-discovered for trending reasons — it gets the lower trending novelty score.

### Fix 4 — Aggregators (Techmeme, Lobsters, etc.) stop laundering 2-week-old essays as fresh

**Symptom**: occasionally an essay from two weeks ago would appear in the brief as if it had just been published, with the aggregator (Techmeme, Lobsters AI) listed as the source.

**Root cause**: aggregator feeds publish `pubDate` based on **when they curated the link**, not when the original article was published. The pipeline was treating that aggregator pubDate as the article's publish time.

**Fix**: aggregator feeds (`techmeme`, `lobsters_ai`, plus `producthunt` which is now disabled outright) are now treated as "signal-only" — they can flag a story for attention, but the publish time is left blank and re-derived from the original article. So no more 2-week-old essays surfacing as if they just hit the wire.

`producthunt` was disabled in the same pass: pubDate-mismatch problem on top of generally low signal-to-noise (most launches don't carry the kind of signal the brief is built for).

### Fix 5 — X channels start flowing again, and faster

**Symptom (the past few days)**: very few X-sourced items in the brief. The X follow-list channel was effectively silent.

**Root cause**: the routine running in the cloud was missing two environment variables (`VERCEL_PROXY_BASE`, `PRISMA_VERCEL_SECRET`) that point at the X cache-warming proxy. Without them, the X fetcher returned an empty list silently — no error, just nothing.

**Fix**: env vars registered in the routine. Local verification confirms the path is restored (the main follow-list channel now returns a normal load of items per slot).

**Bonus — cycle time cut from 138 min to ~70 min**: the X mini-proxy was warming all 83 follow-list handles every cycle, including handles that haven't tweeted in months (`@arena`, `@steipete`, `@ssi`, `@ilyasut`, `@lilianweng`, etc.). Each cold handle ate 12 seconds of cycle budget for nothing. The proxy now tracks **dead handles** — three consecutive empty responses marks a handle as dormant for 24 hours, and a single live response unmarks it. Combined with a batch-size bump from 5 → 7 per cron tick, full-cycle latency dropped from **138 min → ~70 min**. Stabilization takes ~6 hours after deploy as the dead-set populates.

In practice: when a lab official account tweets, the brief sees it within ~70 minutes instead of ~140.

When Meta or Mistral push a new version of their official tooling, the brief catches it within one slot.

### Fix 8 — Old trending repos can no longer lead the brief when the GitHub API is rate-limited (2026-05-12 follow-on)

**Symptom**: the 2026-05-12 10:00 KST brief led with `anthropics/financial-services` framed as if it were a fresh release, but the repo had actually been around since July 2025 — it was simply hitting GitHub's daily-trending list ten months after launch.

**Root cause**: the trending fetcher calls the GitHub REST API to look up each repo's true `created_at`, but unauthenticated calls share a 60 requests/hour cap. Once the cap trips, the fetcher disables enrichment for the rest of that slot and every subsequent repo gets `created_at = null`. The novelty grader, having no age data, fell back to a neutral score — so a 10-month-old repo with a daily-spike got the same novelty treatment as a 2-day-old one.

**Fix**: an "established-repo" heuristic now kicks in when API enrichment fails: **lifetime stars ≥ 5,000 ⇒ treat the repo as older than 90 days** (a 2-week-old repo cannot have accumulated 5,000 stars). Novelty grading then applies the standard >90d cap (score 0), and the repo no longer leads the brief on the strength of a single trending day.

This is a fail-safe — the better long-term fix is to register `GITHUB_TOKEN` in the routine environment so the cap rises to 5,000 requests/hour and the API path stops failing in the first place. The heuristic stays in place either way as a defense in depth.
